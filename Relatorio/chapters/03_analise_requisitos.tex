\section{Estratégia e Métodos}\label{sec:rel_estrategia}

O levantamento de requisitos é uma das fases mais importantes, porque qualquer erro aqui pode causar problemas no desenvolvimento. Para este projeto, foi usada uma abordagem baseada nas boas práticas de engenharia de software \cite{sommerville2015}.

A estratégia usada baseou-se em três pontos:
\begin{enumerate}
    \item \textbf{Usar várias fontes}: Foram usadas várias fontes de informação, como documentos do clube, entrevistas e questionários, para ter uma visão mais completa das necessidades.
    \item \textbf{Consultar os utilizadores}: Foi recolhida informação junto das pessoas que vão usar o sistema, desde a gerência até aos atletas.
    \item \textbf{Revisão dos requisitos}: Sempre que foi identificado um requisito novo, voltou-se a confirmar com o clube se era o que pretendiam.
\end{enumerate}

O primeiro passo foi identificar todos os grupos de pessoas que interagem com o clube, para perceber quem seriam os utilizadores principais do sistema e quem seria afetado de forma indireta.

\begin{itemize}
    \item \textbf{\textit{Stakeholders} primários} (utilizadores diretos do sistema):
    \begin{itemize}
        \item Atletas -- praticantes das diversas modalidades do clube
        \item Treinadores -- responsáveis técnicos das equipas
        \item Encarregados de Educação -- responsáveis legais pelos atletas menores
        \item Secretaria -- funcionários administrativos do clube
        \item Gerência -- órgãos de direção e administração
    \end{itemize}

    \item \textbf{\textit{Stakeholders} secundários} (afetados indiretamente):
    \begin{itemize}
        \item Sócios não-praticantes -- membros associados sem atividade desportiva ativa
        \item Fornecedores e parceiros -- entidades externas com relações comerciais
        \item Autoridades desportivas -- federações e associações reguladoras
        \item Autoridade Tributária -- entidade reguladora fiscal (para efeitos de conformidade SAF-T)
    \end{itemize}
\end{itemize}

Para cada grupo foram identificadas as necessidades e o que esperavam do sistema. Isto ajudou a escolher os métodos de levantamento mais adequados para cada perfil.

\subsection{Planeamento do Levantamento}

O levantamento foi organizado em três fases, com uma duração total de três semanas:

\begin{enumerate}
    \item \textbf{Fase 1 -- Análise Documental} (1 semana): Recolha e análise dos documentos do clube, identificação dos dados e processos existentes.

    \item \textbf{Fase 2 -- Entrevistas} (1 semana): Entrevistas com representantes de cada grupo de utilizadores, sobre problemas atuais e funcionalidades desejadas.

    \item \textbf{Fase 3 -- Questionário e Consolidação} (1 semana): Distribuição do questionário, análise dos resultados e reunião de todos os requisitos num documento preliminar.
\end{enumerate}

\subsection{Análise Documental}\label{sec:rel_documental}

A análise documental foi o ponto de partida do levantamento. Este método permitiu perceber os processos do clube, identificar os dados que o sistema vai ter de gerir e descobrir regras de negócio que não teriam sido identificadas só com entrevistas.

\subsubsection{Objetivo e Abordagem}

O objetivo desta fase foi perceber como a informação circula no clube, identificando:
\begin{itemize}
    \item Os tipos de documentos usados regularmente;
    \item Os dados registados em cada documento;
    \item As relações entre documentos e processos;
    \item As regras de validação que se aplicam;
    \item Os pontos onde o sistema atual falha ou é ineficiente.
\end{itemize}

Foi pedido à gerência do clube exemplos dos documentos que usam no dia a dia. Foram escolhidos três documentos principais que cobrem as três áreas do clube: clínica, desportiva e financeira.

\subsubsection{Documentos Analisados}

\paragraph{Atestado Médico Desportivo}

A análise deste documento foi importante para perceber os requisitos do módulo clínico. O atestado médico é um documento obrigatório para a prática desportiva federada, e sem ele o atleta não pode participar em competições.

Da análise do documento, foram extraídos os seguintes requisitos:
\begin{itemize}
    \item Necessidade de registar a data de emissão e a data de validade do atestado;
    \item Obrigatoriedade de associar cada atestado a um atleta específico;
    \item Requisito de armazenamento do documento digitalizado;
    \item Necessidade de um sistema de alertas automáticos para caducidade (30 dias antes, 7 dias antes, e no dia de expiração);
    \item Bloqueio automático de atletas com atestado caducado para convocatórias e participação em jogos.
\end{itemize}

\verIA{sec:ia_documentos}


\paragraph{Relatório Oficial de Jogo}

O relatório de jogo permitiu perceber os requisitos da área desportiva. Este documento é preenchido manualmente após cada jogo e contém informação sobre o desempenho dos atletas.

Da análise do documento, foram identificadas as seguintes necessidades:
\begin{itemize}
    \item Registo dos atletas titulares e suplentes por jogo;
    \item Acompanhamento dos minutos jogados por cada atleta;
    \item Registo de eventos de jogo: golos marcados, assistências, cartões amarelos e vermelhos;
    \item Resultado final e dados do adversário;
    \item Observações técnicas do treinador.
\end{itemize}

\verIA{sec:ia_documentos}

Estes dados serviram para definir os requisitos do módulo de estatísticas, que vai permitir criar perfis de desempenho para cada atleta e equipa.

\paragraph{Fatura/Recibo de Quota}

A análise da fatura de quotas mostrou os requisitos do módulo financeiro e permitiu perceber como funciona o ciclo de pagamentos no clube.

Da análise do documento, foram identificadas as seguintes necessidades:
\begin{itemize}
    \item Associação de identificadores únicos a cada pagamento;
    \item Registo de datas limite de pagamento por período (mensal, trimestral, anual);
    \item Gestão de estados de pagamento (pendente, pago, em atraso, reembolsado);
    \item Geração automática de faturas em conformidade com requisitos fiscais;
    \item Exportação para formato SAF-T conforme exigido pela Autoridade Tributária.
\end{itemize}

\subsection{Entrevistas Direcionadas}\label{sec:rel_entrevistas}

As entrevistas foram o método principal de levantamento. Foram feitas entrevistas a três tipos de utilizadores do clube: um elemento da Gerência, um Treinador e um Atleta.

\subsubsection{Metodologia das Entrevistas}

Cada entrevista foi dividida em três partes:

\begin{enumerate}
    \item \textbf{Contextualização} (5 minutos): Apresentação do projeto e dos objetivos da entrevista.

    \item \textbf{Questões estruturadas} (20 minutos): Perguntas sobre:
    \begin{itemize}
        \item O que fazem no dia a dia no clube;
        \item Problemas com os processos atuais;
        \item Funcionalidades que gostariam no novo sistema;
        \item O que esperam em termos de facilidade de uso.
    \end{itemize}

    \item \textbf{Discussão aberta} (15 minutos): Espaço para o entrevistado falar sobre outras questões ou sugerir funcionalidades não mencionadas.
\end{enumerate}

As entrevistas foram gravadas em áudio com autorização e depois transcritas. De cada transcrição foram retirados os requisitos identificados, organizados por área funcional.

\subsubsection{Resultados por Perfil de \textit{Stakeholder}}

As entrevistas mostraram que a falta de organização está a causar perdas financeiras e problemas de comunicação. Os principais pontos de cada entrevista são os seguintes:

\subsubsection*{Perspetiva da Gerência}
\begin{itemize}
    \item \textbf{Perda de Investimento e Gestão Financeira:} O clube perdeu o patrocínio de uma empresa de bebidas energéticas simplesmente porque a gerência não conseguiu compilar e apresentar dados estruturados sobre o desempenho das equipas para justificar o investimento. Ficou clara a necessidade de \textit{dashboards} financeiros, gráficos de desempenho exportáveis e geração automática de ficheiros SAF-T.
    \item \textbf{Caos nas Quotas:} O modelo atual de pagamentos manuais resulta em esquecimentos frequentes por parte dos encarregados de educação. É urgente criar um perfil parental agregador com geração de referências Multibanco/MBWay e bloqueio automático de atletas com mensalidades em atraso.
    \item \textbf{Conflitos de Espaços:} Foi relatado um episódio em que as equipas de Futsal e Voleibol tentaram usar o pavilhão ao mesmo tempo devido a um erro num caderno de papel. O sistema tem de gerir o calendário de instalações.
\end{itemize}

\verIA{sec:ia_entrevistas}

\subsubsection*{Perspetiva da Equipa Técnica (Treinadores)}
\begin{itemize}
    \item \textbf{Falhas de Comunicação:} A dependência de grupos de WhatsApp para avisar sobre alterações de horários e convocatórias não funciona. Ocorreram situações em que a equipa foi para jogo com apenas um suplente porque os atletas não viram a mensagem. É obrigatório haver um sistema de convocatórias na plataforma com botões de confirmação (Sim/Não/Talvez).
    \item \textbf{Falta de Histórico Clínico:} Um atleta agravou uma lesão (rotura muscular) durante um treino porque o treinador não foi informado da queixa física que tinha ocorrido em casa. O sistema precisa de um módulo onde a equipa técnica possa consultar os atletas inaptos em tempo real.
    \item \textbf{Estatísticas e Material:} Há a necessidade de gráficos radar para comparar capacidades de atletas da mesma posição, além de um sistema rápido para registar assiduidade, ocorrências de jogo (golos, cartões) e requisitar material à secretaria.
\end{itemize}

\verIA{sec:ia_entrevistas}

\subsubsection*{Perspetiva dos Atletas}
\begin{itemize}
    \item \textbf{Atestados Médicos Caducados:} Um problema crítico identificado foi a falta de avisos sobre a validade dos exames médicos desportivos, o que levou a que um atleta ficasse de fora de uma competição importante. A plataforma tem de alertar automaticamente para a caducidade destes documentos.
    \item \textbf{Autonomia Clínica e Financeira:} Os atletas querem poder registar queixas físicas autonomamente para que estas cheguem de imediato ao treinador, e desejam ter acesso fácil aos horários, ao histórico de pagamentos de quotas e às faturas em formato digital.
    \item \textbf{Perfil Individual Desportivo:} Foi pedida uma área pessoal onde o atleta possa acompanhar a evolução do peso, altura e estatísticas individuais ao longo da época. Foi também manifestado interesse numa biblioteca multimédia para os vídeos táticos partilhados pelo treinador e num espaço para o palmarés (prémios e distinções).
\end{itemize}

\verIA{sec:ia_entrevistas}


\subsection{Questionário Online}\label{sec:rel_questionario}

\subsubsection{Objetivo e Metodologia}

Para complementar as entrevistas e a análise documental, foi feito um questionário online para recolher informação de mais pessoas e perceber quais eram as prioridades da maioria.

O questionário foi criado no Google Forms e enviado para todos os membros do clube com email registado. As respostas foram recolhidas durante uma semana e foram obtidas 47 respostas válidas.

\subsubsection{Estrutura do Questionário}

O questionário foi organizado em quatro partes:

\begin{enumerate}
    \item \textbf{Perfil}: Tipo de utilizador, escalão etário, modalidade(s) praticada(s).

    \item \textbf{Comunicação atual}: Como funciona a comunicação atual (WhatsApp, chamadas, presencial) e com que frequência há falhas de informação.

    \item \textbf{Pagamentos}: Como é feito o pagamento de quotas, se há atrasos, e que métodos de pagamento preferem.

    \item \textbf{Funcionalidades desejadas}: Lista de funcionalidades para priorizar (convocatórias, atestados, reserva de instalações, etc.) e espaço para sugestões adicionais.
\end{enumerate}

\verIA{sec:ia_questionario}



\subsubsection{Análise dos Resultados}

Os resultados confirmaram o que tinha sido dito nas entrevistas:

\begin{itemize}
    \item \textbf{87\%} dos inquiridos classificaram a comunicação atual como ``insuficiente'' ou ``muito insuficiente'';
    \item \textbf{72\%} reportaram pelo menos uma situação nos últimos três meses em que falharam um compromisso por falta de informação atempada;
    \item \textbf{91\%} manifestaram interesse em receber notificações automáticas sobre convocatórias e alterações de horário;
    \item \textbf{65\%} indicaram preferência por pagamento de quotas através de referência Multibanco ou MB Way;
    \item \textbf{78\%} dos encarregados de educação solicitaram acesso a um portal onde possam consultar a situação financeira e os documentos dos seus educandos.
\end{itemize}

Ao juntar estes dados com as entrevistas, foi possível definir uma ordem de prioridades para o desenvolvimento, com os módulos de comunicação e gestão financeira no topo.

\subsection{Síntese e Validação dos Requisitos Levantados}

A combinação das três fontes de informação (documentos, entrevistas e questionário) permitiu ter uma visão completa das necessidades do sistema. Os requisitos foram reunidos num documento que foi apresentado à gerência para validar antes de avançar para a especificação formal.

Nessa reunião foram discutidos os seguintes pontos:
\begin{itemize}
    \item Verificar se todas as áreas funcionais estavam cobertas;
    \item Definir quais as funcionalidades mais urgentes;
    \item Esclarecer dúvidas e requisitos que pareciam contraditórios;
    \item Identificar limitações técnicas e financeiras que pudessem afetar o projeto.
\end{itemize}

A gerência aprovou o documento de requisitos e foi dada autorização para avançar para a especificação. As secções seguintes apresentam os requisitos identificados.

\subsection{User Stories Representativas}\label{sec:user_stories}

Antes da especificação formal, foi feito um resumo da informação das entrevistas e do questionário em \textit{user stories}, no formato \emph{``Como \textless papel\textgreater, quero \textless ação\textgreater, para \textless benefício\textgreater''}. As quinze \textit{stories} mais representativas, organizadas por perfil, são as seguintes:

\paragraph{Atleta / Encarregado de Educação}
\begin{itemize}
    \item Como atleta, quero confirmar a minha disponibilidade para um jogo num clique, para que o treinador saiba com quem pode contar sem precisar de me ligar.
    \item Como encarregado de educação com dois filhos no clube, quero pagar as duas quotas com uma única referência Multibanco, para evitar duplicações e atrasos.
    \item Como atleta, quero receber um alerta 30 dias antes do meu atestado caducar, para não falhar competições por documentação atrasada.
    \item Como encarregado de educação, quero ver num único ecrã o estado financeiro e clínico dos meus educandos, para acompanhar a vida desportiva deles sem ter de ir à secretaria.
    \item Como atleta, quero registar uma queixa física a partir de casa, para que o treinador não me puxe num treino e me agrave a lesão.
\end{itemize}

\paragraph{Treinador}
\begin{itemize}
    \item Como treinador, quero ver imediatamente quais os atletas inaptos (atestado caducado, lesão, quotas em atraso), para não correr o risco de os incluir numa convocatória.
    \item Como treinador, quero remarcar um treino a partir do telemóvel e que os atletas sejam avisados automaticamente, para não aparecerem no campo à hora errada.
    \item Como treinador, quero registar as presenças no telemóvel durante o próprio treino, para não ter de copiar listas em papel mais tarde.
    \item Como treinador, quero comparar atletas da mesma posição através de gráficos radar, para tomar decisões técnicas com base em dados objetivos.
\end{itemize}

\paragraph{Secretaria / Gerência}
\begin{itemize}
    \item Como funcionário da secretaria, quero ter uma fila única de aprovações pendentes (registos, atestados, documentos federativos), para não saltar entre ecrãs durante o dia.
    \item Como gerente, quero exportar o ficheiro SAF-T com um clique no fim do mês, para cumprir as obrigações fiscais sem precisar de contabilista para isso.
    \item Como gerente, quero ver \textit{dashboards} financeiros e desportivos exportáveis em PDF, para apresentar a potenciais patrocinadores em reuniões.
    \item Como gerente, quero bloquear automaticamente atletas com mais de dois meses de quotas em atraso, para que a regra do clube seja aplicada de forma consistente.
    \item Como funcionário da secretaria, quero validar um atestado médico submetido digitalmente sem ter de imprimir o ficheiro, para reduzir o trabalho de arquivo em papel.
    \item Como gerente, quero gerir o calendário de reservas das instalações num único ecrã, para que duas equipas nunca apareçam no mesmo pavilhão à mesma hora.
\end{itemize}

Cada \textit{user story} está associada a um ou mais requisitos funcionais nas secções seguintes.

\subsection{Norma de Especificação Adotada}

Os requisitos seguintes foram especificados segundo as normas \textbf{IEEE 830-1998} e \textbf{ISO/IEC/IEEE 29148:2018}. Cada requisito tem um identificador único, prioridade, pré-condições, pós-condições e critérios de aceitação, e está ligado à sua origem (entrevista, documento ou questionário).

O RF-02 define como os encarregados de educação se ligam aos perfis dos atletas menores. Esta funcionalidade foi pedida na entrevista com a gerência e no questionário, onde 78\% dos encarregados pediram um portal único para acompanhar a situação dos educandos.

Para criar a ligação, os dois perfis têm de já existir e a associação tem de ser confirmada pela secretaria ou por um código do clube. Depois de ligados, as quotas dos educandos são geridas em conjunto, com uma única referência Multibanco/MBWay, e os avisos são enviados para o encarregado. Quando o atleta atingir a maioridade, a ligação pode ser removida.
