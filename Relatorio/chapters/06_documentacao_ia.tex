\section{Abordagem Geral e Ferramentas Utilizadas}\label{sec:ia_abordagem}

Esta secção identifica os modelos usados e a forma como a equipa interagiu com eles.

\subsection{Modelos Utilizados}

Foram usados vários modelos, mas dois foram os mais frequentes:

\begin{itemize}
    \item \textbf{Google Gemini} (família 2.5 Pro e Flash), usado principalmente pela interface Web para tarefas exploratórias: gerar documentos do clube, estruturar o questionário, simular entrevistas e fazer \textit{brainstorming} de opções arquiteturais.

    \item \textbf{Anthropic Claude} (modelos Sonnet 4.5 e 4.6), usado quase sempre via terminal (Claude Code) para tarefas com múltiplos ficheiros: gerar código C\#, fazer \textit{refactoring} e corrigir o relatório inteiro.
\end{itemize}

Em fases mais específicas, recorremos pontualmente a:

\begin{itemize}
    \item \textbf{Vercel v0}, utilizado na fase de \textit{design} de interfaces para gerar protótipos das principais páginas da aplicação (\textit{dashboards} dos quatro perfis de utilizador) a partir de descrições textuais. A diferença para uma resposta normal de LLM é que o \textit{v0} devolve diretamente código React/JSX funcional, que serviu de base para a implementação da camada de apresentação.

    \item Modelos da OpenAI (GPT-4o e GPT-5), pontualmente, sobretudo para comparar respostas em decisões críticas e para confirmar diagnósticos quando o Gemini ou o Claude pareciam estar a alucinar.
\end{itemize}

A combinação Gemini + Claude foi a que melhor funcionou. O Gemini serve para explorar ideias; o Claude no terminal serve para trabalhar diretamente nos ficheiros.

\subsection{Estratégias de Interação: Web vs CLI}

Os modelos foram usados de duas formas distintas.

\paragraph{Interface Web.}
Usada para perguntas pontuais, geração de documentos e estruturação do questionário. A vantagem é a rapidez; a limitação é que o modelo não tem acesso direto aos ficheiros do projeto.

\paragraph{Interface de terminal (Claude Code, Gemini CLI).}
O LLM é executado diretamente no diretório do projeto. É mais lento de preparar, mas oferece vantagens importantes:

\begin{itemize}
    \item \textbf{Acesso direto aos ficheiros.} O modelo lê e edita diretamente ficheiros \texttt{.tex}, \texttt{.cs}, \texttt{.puml}, sem precisar de colar código.
    \item \textbf{Contexto do projeto.} As respostas são mais precisas porque o modelo vê toda a estrutura de diretórios.
    \item \textbf{Operações em \textit{batch}.} Tarefas repetitivas (como gerar 36 tabelas de requisitos) podem ser feitas num único \textit{prompt}.
\end{itemize}

O risco é que o modelo opera diretamente nos ficheiros. A equipa usou \textit{commits} frequentes e revisão dos \textit{diffs} para mitigar isso.

\subsection{Utilização Passiva: LLM como Assistente de Escrita}

Os modelos foram também usados como assistentes de escrita para corrigir gralhas, melhorar frases e validar sintaxe \LaTeX{}. Quando se delegou ao modelo a redação de várias secções seguidas, o texto ficava demasiado uniforme e foi necessário reescrever partes para introduzir variação. Este risco está discutido na Secção~\ref{sec:ia_validacao}.

\section{Fase 1 --- Definição e Contextualização do Sistema}\label{sec:ia_definicao}

Nesta fase não foi usada IA para criar conteúdo. O caso de estudo foi criado pela equipa, aproveitando que um dos elementos é natural de Polvoreira. Os modelos foram usados apenas para rever gramática e corrigir gralhas no texto escrito pela equipa.

\section{Fase 2 --- Elicitação e Análise de Requisitos}\label{sec:ia_requisitos}
\verRelatorio[Capítulo de Análise de Requisitos]{sec:rel_estrategia}

Esta foi a fase onde a IA teve mais peso. Os artefactos produzidos incluem os documentos sintéticos do clube, as entrevistas simuladas, o questionário online e o detalhamento dos requisitos.

\subsection{Definição da Estratégia de Levantamento}

Foi pedida ao Gemini uma lista de métodos de levantamento de requisitos para o contexto do CAP. A resposta foi filtrada para os três métodos que faziam sentido dentro do prazo: análise documental, entrevistas semiestruturadas e questionário online.

\caixaprompt{Gemini (Web)}{No contexto do CAP (Clube Amigos de Polvoreira), sendo um clube desportivo com várias modalidades e diferentes tipos de pessoas (atletas, pais, treinadores, gerência), quero que me digas quais são os melhores métodos para o levantamento de requisitos para o nosso novo sistema de software.}{O modelo sugeriu uma abordagem mista (triangulação de métodos), listando os seguintes métodos como os mais indicados para o caso em estudo:

\begin{itemize}
    \item \textbf{Análise Documental:} Fundamental para perceber ``o que existe''. Devem recolher os atestados médicos, relatórios de jogo em papel e os livros de recibos de quotas atuais para perceber exatamente que dados precisam de ser transpostos para o digital.

    \item \textbf{Entrevistas Direcionadas (Semiestruturadas):} Essencial para captar dores específicas. Recomendo entrevistar representantes chave: um elemento da direção (para a vertente financeira/SAF-T), um treinador (sobre convocatórias e lesões) e um atleta.

    \item \textbf{Questionários Online:} Sendo um clube com muitos associados, a melhor forma de validar problemas à escala (ex: dificuldades de comunicação em grupos de WhatsApp ou métodos de pagamento preferidos) é enviando um formulário via email para todos os atletas e encarregados de educação.

    \item \textbf{Observação Direta (Etnografia):} Passar uma tarde na secretaria do clube num dia de jogo ou de inscrições para observar os estrangulamentos do processo manual.

    \item \textbf{Focus Groups (Grupos de Foco):} Juntar numa sala treinadores de diferentes modalidades para debater como deve funcionar o sistema de reserva do pavilhão sem gerar conflitos.

    \item \textbf{Prototipagem Rápida:} Criar ecrãs falsos (\textit{mockups}) do sistema e mostrar à gerência para garantir que a usabilidade se adequa à literacia digital dos funcionários.
\end{itemize}

Recomendo que escolham 2 a 3 destes métodos para garantir uma boa cobertura sem esgotar o tempo útil do projeto.}

Foi pedida ao Gemini uma listagem categorizada de \textit{stakeholders}. O modelo identificou a maior parte dos perfis, mas \textbf{deixou de fora os Encarregados de Educação}. A equipa acrescentou esse perfil manualmente --- veio a ser um dos mais importantes do sistema (RF-02).

\caixaprompt{Gemini (Web)}{No contexto do CAP (Clube Amigos de Polvoreira), identifica e categoriza todos os \textit{stakeholders} do clube relevantes para o novo sistema, distinguindo entre utilizadores diretos e entidades indiretamente afetadas.}{O modelo dividiu os intervenientes em duas categorias principais, com base na estrutura típica de um clube desportivo português:

\textbf{\textit{Stakeholders} Primários (Utilizadores Diretos):}
\begin{itemize}
    \item \textbf{Atletas:} Os praticantes das diversas modalidades do clube.
    \item \textbf{Treinadores:} Os responsáveis técnicos das equipas, que farão convocatórias e registo de lesões.
    \item \textbf{Secretaria:} Os funcionários administrativos que gerem inscrições, validação de atestados e processos diários.
    \item \textbf{Gerência:} Os órgãos de direção e administração que necessitam de uma visão global financeira e desportiva.
\end{itemize}

\textbf{\textit{Stakeholders} Secundários (Afetados Indiretamente):}
\begin{itemize}
    \item \textbf{Sócios não-praticantes:} Membros associados sem atividade desportiva ativa.
    \item \textbf{Fornecedores e parceiros:} Entidades externas com relações comerciais com o clube.
    \item \textbf{Autoridades desportivas:} Federações e associações reguladoras das competições.
    \item \textbf{Autoridade Tributária (AT):} Entidade reguladora fiscal, essencial para efeitos de conformidade e exportação do SAF-T.
\end{itemize}}

\subsection{Simulação de Entrevistas}\label{sec:ia_entrevistas}
\verRelatorio[Capítulo de Análise de Requisitos]{sec:rel_entrevistas}

Foram simuladas entrevistas a três perfis: gerência, treinador e atleta. É uma simulação, não entrevistas reais. O Gemini foi instruído a usar problemas reais de contexto português (papelada, WhatsApp, atestados caducados). O resultado ficou satisfatório logo no primeiro \textit{prompt}. As três transcrições são reproduzidas abaixo.

\clearpage
\subsubsection*{Entrevista à Gerência}

\caixaprompt{Gemini (Web)}{No contexto do CAP, simula uma entrevista semiestruturada entre um membro da equipa de desenvolvimento e um membro da Gerência do clube. A entrevista deve focar-se nos problemas reais do dia a dia (gestão financeira, perda de patrocínios, comunicação, conflitos de espaços) e nas funcionalidades desejadas para o novo sistema. Produz a transcrição completa em formato pergunta-resposta.}{Transcrição reproduzida integralmente abaixo.}

\begin{tcolorbox}[colback=white, colframe=gray!50, arc=1mm, title=\textbf{Transcrição da Entrevista à Gerência}]
\small
\textbf{Entrevistador:} Olá! Obrigado por nos receber. Estamos a finalizar o levantamento de requisitos para o novo sistema do clube. Do ponto de vista da gerência, quais têm sido as maiores dores de cabeça com o método de trabalho atual?

\textbf{Membro da Gerência:} Olá, bem-vindos. A verdade é que temos imensos processos manuais que nos atrasam e prejudicam o clube. Um dos maiores problemas recentes foi a perda de um potencial investidor, uma empresa de bebidas energéticas. Eu tive uma reunião com eles, mas como não temos uma boa forma de guardar e organizar a informação sobre o desempenho dos atletas e do clube no geral, não consegui comprovar o nosso valor. A empresa acabou por largar o contrato.

\textbf{Entrevistador:} Isso é um impacto financeiro direto. Que ferramentas o novo sistema deve ter para evitar que isso se repita?

\textbf{Membro da Gerência:} É fundamental termos \textit{dashboards} sobre a contabilidade do clube e gráficos de desempenho de todas as modalidades que sejam fáceis de exportar. Precisamos de mostrar a evolução do clube com facilidade e profissionalismo em reuniões com futuros investidores. Além disso, precisamos de gerir melhor os nossos próprios patrocínios e painéis publicitários na mesma plataforma.

\textbf{Entrevistador:} E em relação à gestão do dia a dia com os atletas, especialmente no que toca aos pagamentos das quotas, como avalia a situação atual?

\textbf{Membro da Gerência:} É um caos, para ser sincero. Temos muitos encarregados de educação com filhos em modalidades diferentes. Como os pagamentos são feitos em secretarias diferentes, os pais perdem o controlo, esquecem-se de pagar e nós andamos constantemente a fazer chamadas de cobrança. Precisamos de um sistema que permita criar um perfil parental único, onde os pais vejam as contas consolidadas e possam gerar referências Multibanco ou MBWay. Do nosso lado, o sistema devia ter uma zona com todos os atletas com quotas em atraso e aplicar bloqueios automáticos a quem tem vários meses em dívida.

\textbf{Entrevistador:} Muito claro. Para além da componente financeira, mencionaram problemas logísticos nos inquéritos iniciais. Pode dar-me um exemplo?

\textbf{Membro da Gerência:} Sim, a gestão de espaços físicos. Ainda há pouco tempo, numa noite chuvosa, as equipas de Futsal e de Voleibol entraram no pavilhão ao mesmo tempo porque houve um erro no nosso caderno em papel na receção. Gerou-se um enorme conflito interno. É urgente termos um calendário digital unificado para a reserva de espaços físicos que impeça duplas marcações e até nos permita gerir o aluguer de espaços a entidades externas para gerar receita.

\textbf{Entrevistador:} Para terminar, há mais alguma tarefa burocrática pesada que gostariam de ver automatizada?

\textbf{Membro da Gerência:} Várias. Precisamos de automatizar o processamento de inscrições nas federações e a gestão de seguros desportivos em grupo. Na área da contabilidade, seria uma ajuda tremenda se o sistema gerasse os ficheiros normalizados (SAF-T) para exportação. E, claro, precisamos de uma forma fácil de comunicar com a equipa técnica e os atletas sem depender de dezenas de grupos de WhatsApp.
\end{tcolorbox}

\textbf{Entrevistador:} Para além dessa componente médica, que outras ferramentas de gestão técnica gostava de ter?

\textbf{Treinador:} Gostava de ter acesso a \textit{dashboards} com informação sobre o desempenho da equipa e de cada atleta. \textbf{Entrevistador:} Em termos logísticos e de infraestruturas, há algum processo que falhe regularmente?

\textbf{Treinador:} A reserva de espaços é frequentemente caótica. Precisamos de conseguir fazer a reserva de espaços físicos específicos, como o campo principal, o ginásio ou a sala de vídeo, diretamente numa plataforma digital. Para além disso, daria jeito poder fazer a requisição de material desportivo, como bolas, coletes e pinos, diretamente à gerência para as sessões de treino.

\subsubsection*{Padrão Arquitetural Global}

\caixaprompt{Gemini 2.5 Pro (Web)}{No contexto do CAP --- equipa pequena de quatro estudantes de Engenharia, sistema com obrigações RGPD (dados clínicos) e SAF-T (financeiro), prazo académico de um semestre --- preciso de três opções credíveis para o padrão arquitetural global. \textbf{1. Monólito em Camadas (Layered Monolith).} Aplicação única organizada em camadas (apresentação, negócio, dados).
\textit{Vantagens:} (i) curva de aprendizagem reduzida; (ii) implantação trivial; (iii) transações ACID locais.
\textit{Desvantagens:} (i) acoplamento crescente entre módulos; (ii) dificuldade em isolar dados clínicos do resto do sistema (RGPD); (iii) refactor caro depois de estabilizado.

\textit{Vantagens:} (i) escalabilidade por domínio; (ii) isolamento físico dos dados sensíveis; (iii) equipas pequenas por serviço.
\textit{Desvantagens:} (i) complexidade operacional (orquestração, observabilidade); (ii) transações distribuídas exigem padrões como \textit{saga} --- arriscado para o SAF-T; (iii) custos de infraestrutura desadequados para a dimensão do clube.

\textit{Desvantagens:} (i) exige disciplina de equipa para não violar as fronteiras; (ii) o \textit{bus} de eventos em memória não é durável por defeito; (iii) testes de integração ainda corrim no mesmo processo.

A escolha final --- Monólito Modular --- foi tomada em ata após pesarmos a dimensão da equipa (quatro pessoas, prazo curto) contra a necessidade de isolar os módulos clínico e financeiro. O risco identificado pelo modelo (\textit{bus} de eventos não durável) foi anotado no \textit{backlog} técnico como ``adoção futura do padrão \textit{outbox}'', tal como referido na Secção~\ref{sec:decisoes_arq}.}

\subsubsection*{Sistema de Gestão de Base de Dados}

\caixaprompt{Gemini 2.5 Pro (Web)}{Continuando o exercício anterior, agora para a base de dados. O sistema do CAP tem um módulo financeiro com obrigações ACID estritas (faturas, pagamentos, SAF-T) e um módulo desportivo com métricas semi-estruturadas (avaliações físicas, queixas clínicas livres). Quero três SGBD candidatos, com prós e contras específicos para estes dois requisitos. Não decidas por mim.}{\textbf{1. MongoDB.}
\textit{Vantagens:} esquema flexível ideal para métricas heterogéneas; \textit{queries} sobre documentos aninhados; replicação fácil.
\textit{Desvantagens:} transações multi-documento limitadas e dispendiosas; sem garantia ACID estrita na configuração \textit{default}; geração de relatórios fiscais SAF-T fica frágil sem joins reais.

\textbf{2. Microsoft SQL Server.}
\textit{Vantagens:} ACID estrito; ferramentas maduras de auditoria; \textit{tooling} corporativo conhecido.
\textit{Desvantagens:} custo de licenciamento; \textit{footprint} elevado; menos idiomático em ambiente Linux/Docker.

\textbf{3. PostgreSQL.}
\textit{Vantagens:} ACID estrito; tipo \texttt{jsonb} permite armazenar métricas semi-estruturadas no mesmo motor que o financeiro; gratuito e \textit{open-source}.
\textit{Desvantagens:} indexação de campos \texttt{jsonb} complexos exige planeamento; menos \textit{tooling} visual de auditoria do que o SQL Server por defeito.}

\noindent\textit{\textbf{Nota da equipa:} Curiosamente, o modelo \emph{antes} desta clarificação tinha sugerido MongoDB como a sua primeira escolha, invocando a flexibilidade dos documentos. Quando lhe explicitámos a exigência do SAF-T, recuou e admitiu que a falta de ACID nativo era um risco real. Esta inversão reforçou a nossa convicção de que o \textit{prompt} tem de carregar as restrições não negociáveis logo de início --- caso contrário o modelo otimiza para a métrica errada (flexibilidade em vez de integridade).}

\subsubsection*{\textit{Stack} de \textit{Backend}}

\caixaprompt{Gemini 2.5 Pro (Web)}{Para o \textit{backend} do CAP, dá-me três opções de \textit{stack}, considerando que precisamos de \textit{cronjobs} para vigiar caducidades de atestados, gerar SAF-T em lote e emitir notificações em diferido. Atenção: a equipa tem mais experiência em C\# do que em Java ou JavaScript. Inclui esse fator na análise.}{\textbf{1. Node.js com NestJS.}
\textit{Vantagens:} arranque rápido; ecossistema NPM amplo; modelo assíncrono natural para \textit{webhooks}.
\textit{Desvantagens:} \textit{single-threaded} --- a geração do XML do SAF-T pode bloquear o \textit{event loop}; gestão de \textit{cronjobs} típica via \texttt{node-cron} é menos robusta que alternativas hospedadas.

\textbf{2. Java com Spring Boot.}
\textit{Vantagens:} ecossistema corporativo maduro; \textit{tooling} robusto; suporte transacional sólido.
\textit{Desvantagens:} verbosidade; \textit{footprint} de memória elevado; curva mais íngreme se a equipa tem mais experiência em C\#.

\textbf{3. C\# com .NET 7.}
\textit{Vantagens:} \texttt{IHostedService}/\texttt{BackgroundService} integrados no próprio \textit{runtime} (ideal para \textit{cronjobs}); Entity Framework Core integra-se naturalmente com PostgreSQL; performance comparável ao Java; idiomático para a equipa.
\textit{Desvantagens:} ecossistema mais fechado historicamente (embora \textit{open-source} desde .NET Core); algumas dependências de \textit{tooling} ainda preferem ambiente Windows.}

\noindent\textit{\textbf{Nota da equipa:} A escolha pareceu óbvia após a resposta, mas vale registar uma alucinação benigna: o modelo afirmou inicialmente que o EF Core ``não suporta nativamente PostgreSQL'', o que é falso desde 2017 (o \textit{provider} \texttt{Npgsql.EntityFrameworkCore.PostgreSQL} é maduro e amplamente usado). Confrontado com a documentação oficial, o modelo retratou-se. Foi um lembrete útil de que mesmo em terreno conhecido o modelo pode inventar limitações que não existem --- é por isso que para cada afirmação técnica sensível confirmámos com a documentação canónica antes de avançar.}

\subsubsection*{Paradigma de Comunicação e API}

\caixaprompt{Gemini 2.5 Pro (Web)}{Por fim, o paradigma de API. Considera: clientes Web (Next.js) e \textit{webhooks} HMAC vindos de \textit{gateways} de pagamento Multibanco/MBWay; RBAC granular por papel. Dá-me três paradigmas, com prós e contras alinhados a estas realidades.}{\textbf{1. GraphQL.}
\textit{Vantagens:} cliente decide o \textit{shape} dos dados; uma única \textit{endpoint}; \textit{schema} fortemente tipado.
\textit{Desvantagens:} RBAC tem de ser aplicado em cada \textit{resolver}/campo, o que é propenso a erros; \textit{webhooks} externos não falam GraphQL.

\textbf{2. gRPC.}
\textit{Vantagens:} eficiência binária; \textit{streaming} bidirecional; contrato fortemente tipado via \textit{protobuf}.
\textit{Desvantagens:} suporte limitado nos \textit{browsers} sem \textit{proxy}; integração com \textit{webhooks} HTTP/JSON desajeitada; curva de aprendizagem elevada.

\textbf{3. REST + JSON.}
\textit{Vantagens:} ubíquo; RBAC aplicável por rota com \textit{middleware} simples; \textit{webhooks} são naturalmente HTTP/JSON.
\textit{Desvantagens:} \textit{over-fetching} e \textit{under-fetching} se as rotas não forem desenhadas com cuidado; sem contrato formal nativo (mitigável com OpenAPI/Swagger).}

\noindent\textit{\textbf{Nota da equipa:} Aqui o modelo subestimou a complexidade do RBAC em GraphQL --- só após insistirmos é que reconheceu que a autorização por nó do grafo é um ponto de fricção real. A análise final que ficou registada na Tabela~\ref{tab:comparacao_api} do Capítulo~\ref{chap:especificacao_modelacao} reflete já esta correção introduzida pela equipa.}

\subsubsection*{O que correu bem, o que correu mal}

A estratégia de três opções por tópico funcionou bem: forçou o modelo a apresentar contrapontos em vez de defender a sua escolha, e produziu respostas facilmente transponíveis para tabelas.

Foram identificadas três limitações recorrentes:

\begin{itemize}
    \item \textbf{Tendência para o \textit{hype}.} Sem restrições explícitas, o modelo escolhia opções mais complexas (microsserviços, GraphQL, MongoDB) sem considerar a dimensão do problema.
    \item \textbf{Alucinações em terreno conhecido.} O modelo afirmou que o EF Core ``não suporta PostgreSQL'', o que é falso.
    \item \textbf{Contras genéricos.} Os pontos negativos eram vagos e tiveram de ser especificados em termos do contexto real do CAP.
\end{itemize}

\subsection{Geração da Matriz de Permissões RBAC}\label{subsec:ia_arq_rbac}
\verRelatorio[Capítulo de Especificação e Modelação, secção da Matriz RBAC]{sec:rel_rbac}

A Matriz de Permissões RBAC (reproduzida na Tabela~\ref{tab:rbac} do Capítulo~\ref{chap:especificacao_modelacao}) é um dos artefactos com maior impacto na segurança do sistema: cada linha mapeia diretamente em pelo menos um atributo \texttt{[Authorize(Roles = "...")]} dos \textit{controllers} ASP.NET Core. Um erro de atribuição traduz-se, na prática, num \textit{endpoint} aberto a quem não devia ter acesso, ou inacessível a quem devia. Por esta razão, a geração foi feita num único \textit{prompt} ao Claude Code com contexto total dos requisitos, mas a revisão foi exaustiva.

\caixaprompt{Claude Sonnet 4.5 (Claude Code)}{Lê o ficheiro \texttt{chapters/03\_analise\_requisitos.tex} onde estão documentados todos os requisitos funcionais (RF-01 a RF-23) e os papéis de utilizador identificados (Atleta, Encarregado de Educação, Treinador, Secretaria e Gerência). Gera uma tabela LaTeX com a Matriz de Permissões RBAC completa: linhas são as operações principais do sistema (uma ou mais por RF) e colunas são os papéis. Usa os símbolos \textbackslash checkmark (acesso permitido) e --- (sem acesso). Para cada operação, fundamenta a atribuição na regra de negócio descrita no respetivo RF.}{Matriz gerada com 23 operações cruzadas com os 5 papéis. As atribuições foram derivadas das regras de negócio explícitas nos RF: Treinador com acesso completo ao módulo desportivo; Secretaria com acesso ao módulo administrativo e clínico mas sem acesso financeiro estratégico; Gerência com leitura global e acesso exclusivo a relatórios SAF-T e auditoria.}

\paragraph{Critério de revisão.}
A revisão foi feita por dois membros da equipa em conjunto, percorrendo cada uma das 23 linhas e, para cada coluna, respondendo à pergunta: \emph{``Que regra de negócio dos RF justifica este \checkmark{} ou este ---?''}. Sempre que a justificação não era óbvia, a célula era marcada como duvidosa e validada com o RF correspondente. Foram detetados quatro erros de atribuição:
\begin{itemize}
    \item \textit{\textbf{RF-19 (SAF-T) atribuído à Secretaria.} A geração de ficheiros SAF-T é uma obrigação fiscal estratégica e foi reservada exclusivamente à Gerência, conforme documentado na entrevista à Gerência. Corrigido: \textbf{Gerência} \checkmark, \textbf{Secretaria} ---.}
    \item \textit{\textbf{RF-11 (Gestão de EMD) sem acesso ao Encarregado.} Os atestados médicos de menores são geridos pelos encarregados de educação. O modelo não captou esta relação parental. Corrigido: \textbf{Encarregado} (\checkmark).}
    \item \textit{\textbf{RF-06 (Convocatórias) com acesso de leitura ao Atleta omitido.} O atleta precisa de consultar a convocatória para poder responder (RF-07). Acesso de leitura adicionado.}
    \item \textit{\textbf{RF-16 (Loja) erroneamente incluído na Gerência.} A Gerência não gere o inventário da loja diretamente; esse papel cabe à Secretaria. Corrigido.}
\end{itemize}

\paragraph{Padrões observados nos erros.}
Os quatro erros não foram aleatórios; partilhavam duas tendências do modelo. Primeiro, uma \emph{tendência hierárquica}: o LLM presumiu que a Gerência herda automaticamente o acesso de todos os outros papéis e que a Secretaria herda o da operação. Esta presunção é razoável em sistemas administrativos clássicos, mas no CAP a relação não é hierárquica --- é funcional, com a Gerência responsável por decisões estratégicas (SAF-T, contabilidade global) e a Secretaria responsável pelas operações do dia a dia. Segundo, uma \emph{ausência de pensamento sobre o ator parental}: o modelo tratou o Encarregado de Educação como um papel autónomo, sem capturar que herda permissões dos seus educandos (logado como Encarregado age sobre os atestados, quotas e queixas dos atletas associados).

\paragraph{Validação cruzada com os controladores.}
Após a correção manual, foi feita uma segunda passagem: para cada \textit{controller} já implementado, comparou-se o atributo \texttt{[Authorize(Roles = "...")]} com a linha correspondente da matriz. Esta passagem detetou três \textit{controllers} (entre os 16 já implementados na altura) onde o atributo estava em falta --- esses casos estão documentados nas correções ao \textit{backend} apresentadas mais à frente, na Subsecção~\ref{subsec:ia_impl_backend}. A revisão linha a linha demorou cerca de 30 minutos, mas garantiu que o RBAC implementado em C\# refletia as regras de negócio e não a interpretação inicial do modelo.

\subsection{Verificação Cruzada entre Modelos}\label{subsec:ia_arq_crossverification}

Para as decisões mais críticas --- nomeadamente o padrão arquitetural global e a escolha do SGBD --- a equipa adotou uma metodologia de \textbf{verificação cruzada entre modelos}, descrita em três passos:

\begin{enumerate}
    \item \textbf{Mesmo \textit{prompt}, modelos independentes.} O \textit{prompt} é submetido ao Gemini (sessão Web) e ao Claude (sessão separada, sem qualquer memória da conversa do Gemini). As respostas são guardadas em ficheiros separados.
    \item \textbf{Diff de raciocínios.} A equipa lê as duas respostas lado a lado, identificando: convergências (pontos onde ambos concordam), divergências (recomendações opostas) e omissões (ponto referido por um modelo mas não pelo outro).
    \item \textbf{Auditoria recíproca.} As omissões e divergências são depois submetidas como contraprova ao modelo \emph{oposto} (``o outro modelo apontou X como desvantagem; qual é a tua avaliação?''). Esta passagem força cada modelo a defender ou rever a sua posição inicial, e é frequentemente onde aparecem as nuances mais úteis.
\end{enumerate}

A subsecção que se segue ilustra o método aplicado à decisão arquitetural global. O passo 1 corresponde à submissão do mesmo \textit{prompt} ao Claude que já tinha sido submetido ao Gemini na Subsecção~\ref{subsec:ia_arq_decisoes} (\textit{prompt} reproduzido aqui para facilitar leitura). O passo 2 corresponde aos pontos listados após a caixa. O passo 3 corresponde à \emph{auditoria recíproca} que fecha a subsecção.

\caixaprompt{Claude Sonnet 4.5 (Web)}{No contexto do CAP --- equipa de quatro estudantes, prazo académico de um semestre, sistema com dados clínicos (RGPD) e obrigações fiscais SAF-T --- apresenta três opções para o padrão arquitetural global. Para cada uma indica três vantagens e três desvantagens específicas para este contexto. Não escolhas por mim.}{Os três padrões mais adequados ao vosso contexto são o Monólito em Camadas, o Monólito Modular e os Microsserviços. O Monólito em Camadas é a opção de menor risco operacional para uma equipa pequena, mas o acoplamento entre domínios (clínico, financeiro, desportivo) pode tornar a conformidade RGPD difícil de auditar. O Monólito Modular oferece o melhor equilíbrio: fronteiras explícitas por domínio (auditável para RGPD), transações ACID locais (seguras para SAF-T) e implantação simples. \noindent\textit{\textbf{Passo 2 --- diferenças identificadas em relação à resposta do Gemini:}
\begin{itemize}
    \item O \textbf{Claude recomendou explicitamente} o Monólito Modular sem ser questionado; o Gemini recusou dar uma recomendação antes de ser explicitamente instruído a não decidir (o que acabou por ser útil para forçar a análise).
    \item O Gemini identificou como desvantagem do Monólito Modular o facto de o barramento de eventos ser não durável por defeito; o Claude não mencionou esse ponto. A equipa considerou esta omissão relevante e incorporou-a na análise.
    \item Ambos os modelos concordaram em rejeitar os Microsserviços para este contexto --- o que deu à equipa confiança adicional na decisão.
\end{itemize}}

\caixaprompt{Claude Sonnet 4.5 (Web) --- Passo 3, auditoria recíproca}{Outro modelo (Gemini) apontou como desvantagem do Monólito Modular o facto de o barramento de eventos ser, por defeito, não durável (mensagens em memória, perdem-se se o serviço cair). Concordas com esta observação no contexto do CAP? Se sim, que mitigação recomendas que seja compatível com a estratégia de Monólito Modular?}{Concordo. A observação é correta e relevante: num Monólito Modular implementado em ASP.NET Core, o \texttt{MediatR} ou um \texttt{EventBus} em memória \emph{perde} eventos publicados se o processo cair entre o \texttt{publish} e o \texttt{handler}. No contexto do CAP, isto é particularmente sensível para o motor de notificações (eventos como ``convocatória publicada'' não podem perder-se) e para o módulo financeiro (eventos de confirmação de pagamento). Mitigações compatíveis: (i) \textbf{outbox \textit{pattern}} --- gravar o evento na mesma transação da operação de domínio e despachar a partir da tabela \texttt{Outbox} num \textit{worker}; (ii) introduzir um \textit{broker} leve (RabbitMQ ou Redis Streams) só para os eventos críticos, mantendo o resto em memória. Para a vossa dimensão, a opção (i) é provavelmente mais simples e suficiente.}

\noindent\textit{\textbf{Resultado da auditoria recíproca:}} a opinião pedida ao Claude sobre a observação do Gemini não só confirmou o risco como ofereceu uma mitigação concreta (\textit{outbox pattern}) que nenhum dos modelos tinha referido nas respostas iniciais. Esta mitigação ficou registada no Capítulo~\ref{chap:especificacao_modelacao} como decisão arquitetural a aplicar nos pontos críticos. Pedimos depois a confirmação inversa --- isto é, submetemos ao Gemini a recomendação do \textit{outbox pattern} feita pelo Claude. O Gemini validou-a, acrescentando que a \texttt{Outbox} deve ser indexada pela combinação \texttt{(despachado, criadoEm)} para que o \textit{worker} possa fazer \textit{polling} eficiente. Esta nota adicional foi também incorporada na decisão final.

A metodologia de verificação cruzada acabou por ser aplicada também ao paradigma de API (REST vs.\ GraphQL vs.\ gRPC) e à escolha de SGBD (PostgreSQL vs.\ MySQL vs.\ MongoDB). Em todos os casos, o passo 3 (auditoria recíproca) revelou pelo menos um detalhe não capturado pela leitura inicial das duas respostas em paralelo --- o que confirmou o valor de não nos limitarmos à comparação ``A disse X, B disse Y''.

\caixaprompt{Gemini 2.5 Pro (CLI)}{Para cada um dos sete módulos, gera o \texttt{DbContext} correspondente em \texttt{src/Modules/\{Modulo\}/Data/}. \noindent\textit{\textbf{Nota da equipa --- correções aplicadas:} Duas correções importantes:}
\begin{itemize}
    \item \textit{\textbf{Problema N+1.} A primeira versão de \texttt{GetEncarregadoWithDependentesAsync} foi gerada sem \texttt{Include(e => e.AtletasDependentes)}. \caixaprompt{Gemini 2.5 Pro (CLI)}{Configura a injeção de dependências em \texttt{Program.cs}: regista todos os \texttt{DbContext} com \texttt{AddDbContext} a usar a \textit{connection string} \texttt{DefaultConnection}, regista os repositórios com ciclo de vida \texttt{Scoped}, ativa autenticação JWT a partir da configuração, configura Swagger e adiciona política CORS para \texttt{http://localhost:3000}. \noindent\textit{\textbf{Nota da equipa --- correções aplicadas:} O ficheiro resultante era \textbf{tecnicamente correto mas estruturalmente intratável}: aproximadamente 180 linhas só de chamadas a \texttt{builder.Services.Add\ldots}, sem qualquer agrupamento por módulo. \subsubsection*{Autenticação: JWT, BCrypt e \textit{Lockout}}

\caixaprompt{Gemini 2.5 Pro (CLI)}{Implementa o \texttt{AuthController} em \texttt{src/Modules/Users/Controllers/}. Endpoints: \texttt{POST /api/users/auth/login} e \texttt{POST /api/users/auth/register}. No login: valida credenciais com \texttt{BCrypt.Verify}, incrementa \texttt{FailedLoginAttempts} em caso de falha, bloqueia a conta por 15 minutos após 5 falhas consecutivas, gera token JWT com \textit{claims} de \texttt{NameIdentifier}, \texttt{Name}, \texttt{Email} e \texttt{Role}, com expiração de 7 dias. Usa \texttt{HmacSha256Signature}.}{\texttt{AuthController} criado. Login implementado com lógica de bloqueio; registo cria utilizador em estado \texttt{Pendente} aguardando aprovação da secretaria.}

\noindent\textit{\textbf{Nota da equipa --- correções aplicadas:} Aqui acumularam-se \emph{quatro} correções com impacto direto na segurança da aplicação. Vale a pena listá-las porque mostram o tipo de revisão que tivemos de fazer em \emph{todos} os \textit{controllers} sensíveis:}
\begin{itemize}
    \item \textit{\textbf{Reset do contador no sucesso.} O método de \textit{login} incrementava o contador de tentativas falhadas no caminho de erro mas \textbf{não o repunha a zero no caminho de sucesso}. Resultado: um utilizador que se enganasse quatro vezes consecutivas e acertasse à quinta ficava com o contador a 4 e podia ser bloqueado por \emph{um único} erro futuro. Acrescentámos o reset do contador e do \textit{lockout} no caminho de sucesso.}
    \item \textit{\textbf{\textit{Timing attack} no método de procura por email.} A versão gerada devolvia imediatamente \texttt{Unauthorized} se o email não existisse, sem nunca chamar a função de verificação do BCrypt. Isto cria um canal lateral de \textit{timing} que permite enumerar emails válidos: a resposta com email existente demora cerca de 80 milissegundos e a resposta com email inexistente demora cerca de 2 milissegundos. Introduzimos uma chamada à verificação do BCrypt com hash sentinela para o caso de utilizador inexistente, igualando o tempo de resposta.}
    \item \textit{\textbf{BCrypt sem \texttt{WorkFactor} explícito.} O modelo gerou a chamada de \textit{hashing} sem segundo argumento. O \textit{default} da biblioteca é atualmente 11, o que é razoável \emph{hoje}, mas pode mudar entre versões. Tornámos o \texttt{WorkFactor} explícito (valor 12) de forma a fixar a política de \textit{hashing} e a permitir auditoria futura.}
    \item \textit{\textbf{Token sem \textit{audience}/\textit{issuer}.} A primeira versão do \texttt{SecurityTokenDescriptor} omitia \texttt{Issuer} e \texttt{Audience}. Sem esses campos a validação no \textit{middleware} fica laxa --- qualquer JWT assinado com a chave correta é aceite. Adicionámos os dois campos a partir da configuração \texttt{Jwt:Issuer} e \texttt{Jwt:Audience}, e ligámos a validação no \texttt{TokenValidationParameters}.}
\end{itemize}

\subsubsection*{\texttt{Controllers} de Domínio em Bloco}

\caixaprompt{Gemini 2.5 Pro (CLI)}{Cria os 17 \textit{controllers} restantes (ver Tabela~\ref{tab:controladores}), distribuídos pelos sete módulos. Cada um deve usar o respetivo repositório por injeção, expor as operações CRUD necessárias com verbos HTTP corretos e aplicar \texttt{[Authorize(Roles = \ldots)]} de acordo com a Matriz RBAC. Onde for relevante (por exemplo \texttt{ConvocatoriasController}), usa as operações orientadas a domínio definidas na especificação inicial da API (Secção~\ref{subsec:ia_arq_api}).}{17 \textit{controllers} criados, distribuídos pelos sete módulos. Rotas alinhadas com a Tabela~\ref{tab:controladores}. Atributos \texttt{[Authorize]} aplicados.}

\noindent\textit{\textbf{Nota da equipa --- correções aplicadas (síntese):} O pedido em bloco economizou várias horas de \textit{boilerplate}, mas o resultado teve de ser revisto \textit{controller} a \textit{controller}. As categorias de erro recorrentes foram:}
\begin{itemize}
    \item \textit{\textbf{DTOs duplicados.} Cada módulo gerou os seus próprios \textit{requests} de criação e atualização para entidades equivalentes (por exemplo, o pedido de criação de um atleta apareceu duplicado entre os módulos Users e Sports). Movemos os DTOs partilháveis para \ttsplit{CAP.Shared/Contracts/} e ajustámos as referências.}
    \item \textit{\textbf{\texttt{WebhooksController} sem validação HMAC.} Esta foi a correção mais crítica de toda a fase. O \textit{webhook} de confirmação de pagamentos da Ifthenpay envia uma assinatura HMAC-SHA256 no cabeçalho \texttt{X-Signature}. A versão gerada aceitava qualquer pedido vindo do exterior e marcava a quota como paga sem verificar a assinatura --- \textbf{qualquer pessoa com a URL do webhook podia marcar quotas arbitrárias como pagas}. Implementámos a validação HMAC contra a chave partilhada armazenada em \texttt{Configuration["Payments:WebhookSecret"]}, comparando com \texttt{CryptographicOperations.FixedTimeEquals} para evitar \textit{timing attacks}.}
    \item \textit{\textbf{\texttt{[Authorize]} omisso em rotas internas.} Vários \textit{endpoints} de consulta (ex.: \texttt{GET /api/users/athletes}) saíram sem qualquer atributo \texttt{[Authorize]}, o que os deixava acessíveis sem token. Acrescentámos os atributos cruzando manualmente cada rota com a Matriz RBAC.}
    \item \textit{\textbf{Verbos HTTP incorretos.} O \texttt{ConvocatoriasController} gerou corretamente o \textit{endpoint} de publicar, mas para o cancelamento criou uma rota com o motivo no \textit{path} em vez de no \textit{body} (algo como \ttsplit{/calls/\{id\}/cancelar/\{motivo\}}), o que expõe a justificação do cancelamento nos \textit{logs} de servidor. Movemos o motivo para o \textit{body} do pedido.}
\end{itemize}

\subsection{Contentorização com Docker Compose}\label{subsec:ia_impl_docker}

O \texttt{docker-compose.yml} e os dois \texttt{Dockerfile} foram gerados com o Gemini CLI num único \textit{prompt}.

\caixaprompt{Gemini 2.5 Pro (CLI)}{Cria \texttt{docker-compose.yml} na raiz do repositório com três serviços: \texttt{db} (PostgreSQL 15 Alpine), \texttt{api} (ASP.NET Core 7 construído com \textit{multi-stage build}, expondo a porta 5000) e \texttt{frontend} (Next.js 16 construído com \textit{standalone output}, expondo a porta 3000). \texttt{api} depende de \texttt{db}. \noindent\textit{\textbf{Nota da equipa --- correções aplicadas:}}
\begin{itemize}
    \item \textit{\textbf{\textit{Race condition} no arranque.} A versão gerada usava \texttt{depends\_on} sem \texttt{condition: service\_healthy}. Como o contentor da BD demora alguns segundos a estar pronto, a API arrancava antes e falhava na primeira \texttt{Database.Migrate()} com \textit{connection refused}.     \item \textit{\textbf{Segredos em texto limpo.} As \textit{passwords} do PostgreSQL e a \textit{Jwt:Key} estavam codificadas diretamente no \texttt{docker-compose.yml}. Movemos para o ficheiro \texttt{.env} (não versionado) e mantivemos apenas um \texttt{.env.example} no Git.}
\end{itemize}

\subsection{Balanço da Fase 4}\label{subsec:ia_impl_balanco}

Sem o Gemini CLI e o v0, não teria sido possível chegar ao mesmo grau de cobertura funcional dentro do prazo. Os sete \texttt{DbContext}, os dezoito \textit{controllers} e os esqueletos dos \textit{dashboards} foram gerados muito mais rápido do que seria possível manualmente.

Por outro lado, esta foi a fase onde mais se notou que \textbf{a IA não tem juízo de segurança}. As correções desta secção não são polimento: são erros que, em produção, teriam consequências reais (enumeração de utilizadores, \textit{webhooks} sem HMAC, \textit{lockout} mal reiniciado). Nenhum foi detetado pelo modelo --- foram encontrados por revisão humana e testes adversos.

Cerca de \textbf{60--70\% das linhas de código} nasceram num LLM, mas só \textbf{cerca de 20\%} sobreviveram sem alterações. A IA pagou-se em \textit{boilerplate}, mas exigiu revisão atenta em tudo o que tem implicações de segurança.

\subsection{Notas Específicas por Modelo}\label{subsec:ia_prompts_modelos}

Foram também registadas diferenças entre o Gemini (família 2.5) e o Claude (família Sonnet 4.5/4.6):

\paragraph{Gemini.} Bom para \textit{brainstorming} e conteúdo estruturado. Tende a ser verboso e usar negrito em excesso. Em CLI é competente em código C\#.

\paragraph{Claude.} Mais conservador e menos verboso. Em Claude Code opera melhor sobre múltiplos ficheiros com menos alucinações. Tende a oferecer ``depende'' em vez de recomendar diretamente, o que no contexto deste projeto foi mais virtude do que defeito.

\paragraph{v0.} Especializado em componentes React/JSX com pré-visualização visual. Os erros são previsíveis: dados \textit{hardcoded}, strings em inglês, acessibilidade fraca. Útil só em prototipagem visual.

A escolha do modelo certo para cada tarefa foi mais importante do que a qualidade dos \textit{prompts}.

\subsection{O Processo de Revisão da Equipa}\label{subsec:ia_val_processo}

A equipa adotou as seguintes práticas de revisão:

\begin{itemize}
    \item \textbf{Nenhum \textit{commit} sem revisão do \textit{diff}.} Pelo menos um elemento da equipa revia o \textit{diff} antes de fazer \texttt{git add}.

    \item \textbf{Revisão por pares para \textit{controllers} sensíveis.} \textit{Controllers} de segurança (autenticação, \textit{webhooks}, SAF-T) foram revistos por um segundo elemento. Foi assim que se detetou a falha do HMAC.

    \item \textbf{Testes manuais adversos.} Sem testes automatizados extensivos, foram feitas sessões de teste manual: cinco \textit{logins} errados seguidos, \textit{webhook} com assinatura inválida, dois registos simultâneos do mesmo NIF. Detetou-se aqui o problema do contador de \textit{lockout} e o \textit{race condition} do Docker.

    \item \textbf{\textit{Logs} SQL em modo verbose.} Foi assim que se detetou o problema N+1 do repositório do EE: 13 \textit{queries} em vez de uma.

    \item \textbf{Segundo modelo a auditar.} Após gerar um \textit{controller}, foi pedida uma revisão a um modelo diferente. Funcionou bem para erros de segurança.
\end{itemize}

\subsection{Estimativa de Impacto: O Que Ganhámos e O Que Perdemos}\label{subsec:ia_val_impacto}

As estimativas são qualitativas, sem \textit{baseline} comparativa.

\textbf{O que ganhámos.} O ganho principal foi em \textbf{velocidade}. As 36 tabelas de requisitos, os 7 esqueletos C\#, os 18 \textit{controllers} e os componentes React foram gerados muito mais rápido do que seria possível manualmente. A redução de tempo estimada nos artefactos rondou \textbf{50--60\%}. O \textit{brainstorming} com um interlocutor disponível 24/7 também acelerou as decisões arquiteturais.

\textbf{O que perdemos.} O custo principal foi a \textbf{revisão}, muitas vezes subestimada. Para cada hora poupada na geração, gastaram-se cerca de \textbf{10--20 minutos} em revisão, dando um ganho líquido de \textbf{40--50\%}. A uniformização do estilo do relatório foi outro custo: texto gerado em bloco fica demasiado homogéneo e foi necessário reescrever passagens. Por último, a tentação de aceitar uma resposta plausível sem verificar foi constante, especialmente em áreas como segurança e Docker.

\subsection{Conclusão: A IA como Acelerador, Não como Decisor}\label{subsec:ia_val_conclusao}

A IA generativa é um acelerador, mas não substitui o juízo humano nos pontos críticos: decisões arquiteturais, semântica de domínio, segurança. Os ganhos são reais; os riscos também. A diferença entre tirar partido positivo da IA e ser enganado pelos seus erros silenciosos está na disciplina de revisão.

Se o projeto fosse começar de novo, a equipa voltaria a usar IA, mas com três mudanças: (i) reservar tempo explícito para revisão de \textit{diffs}; (ii) adotar testes automatizados desde o início; (iii) usar um segundo modelo para auditar o \textit{output} do primeiro, especialmente em decisões de segurança.
